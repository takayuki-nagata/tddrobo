- **Dynamic Oracle Verification**: When designing test cases that involve calculations, register state changes (e.g., scale, ibase, obase), or radix conversions, you MUST populate the structural verification fields (`oracle_expression` and `oracle_expected`) so the verification engine can check the behavior against the dynamic oracle.
  - **Oracle Backend**: The verification engine executes the `oracle_expression` directly within a standard POSIX `bc` calculator backend. Your expression must conform to POSIX `bc` syntax.
  - **`oracle_expression`**: A semicolon-separated string of statements that sets up the registers and performs the final calculation/verification. It MUST include the state setup and the target evaluation expression (e.g., 'scale=5; scale' or 'ibase=16; obase=8; 20'). Do NOT write assertion words or descriptions.
  - **`oracle_expected`**: The exact expected raw output string evaluated from the `oracle_expression` (e.g., '5', '16', '4.00', '.45').
    - **CRITICAL WARNING**: Do NOT include any descriptive natural language or assertion words (e.g. do NOT write "Prints 4.00", "Outputs .45", or "Returns 0"). It MUST be the absolute raw string representation returned by the execution, otherwise verification will fail.
    - **Empty Output Case**: If the `oracle_expression` yields no standard output in POSIX `bc` (such as comment blocks `/* ... */`, empty lines, or variable assignments without a printing expression), `oracle_expected` MUST be an empty string `""`.
  - **Fallback**: For non-computational or generic test cases where oracle verification is not applicable, you may leave these fields as `null` or omit them.
